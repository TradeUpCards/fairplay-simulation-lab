Scoring Architecture — what model does what
Every score the demo needs, the model behind it, whether it has a champion/challenger pair, and how they depend on each other. The throughline: ML competes in perception, never in the decision.
8 scores 3 with champion/challenger 1 frozen-policy (no ML) Eval lab = the selection harness
§01

Perception vs. Decision — the line that sets every model choice

~3 min

Before picking models, draw one line. Perception scores estimate what is true about a player or table (their type, their fit, the table's health, a cluster's risk). Decision scores choose what to do (rank tables, gate a seating). The model policy follows directly:

Perception → ML is allowed

Estimate truth from features

Classification, Fit, predation pressure, integrity scoring. Here a model legitimately beats a hand-tuned rule, so this is exactly where a champion/challenger pairing makes sense — a simple baseline that ships, and a trained interpretable model that competes in the eval lab.

Decision → deterministic only

Choose an action from scores

The router Rank(P,T) and the integrity hard-gate. No model here, ever. A frozen, human-designed weighted formula — auditable, explainable, overridable. There is no "challenger model" for the decision; you calibrate weights, you don't swap a black box in.

So the answer to "how many models?" — three perception scores carry a real champion/challenger pair (Classification, Fit, the Ppred health term). The rest are deterministic formulas or graph algorithms you calibrate, not models you swap. The decision layer is frozen by mandate.

§03

The Champion / Challenger Framework

~5 min

Champion/challenger means: a champion is live and always the fallback; one or more challengers run in shadow and only get promoted if they beat the champion on a held-out, labeled metric. For us the eval lab is the comparison harness — it already has seeded ground truth and the "true-risk ranks above false-positive traps" invariant. That makes it the natural place to score a challenger against a champion.

● Champion — ships the demo
Deterministic baseline
A transparent rule or formula. Always works, always auditable, never blocks the demo. For Classification it's literally the threshold rules P2 already wrote into the data note. This is what's live on Day 2.
● Challenger — the upgrade
Trained interpretable model. Logistic regression or gradient-boosted trees with feature-importance as reason codes. Runs in the eval lab against the champion. Promoted only if it (a) beats the champion on labeled accuracy AND (b) stays interpretable. Wins → it becomes champion; loses → champion holds.
Where champion/challenger does NOT apply: the deterministic health terms (Pfrag, Pbleed) — you calibrate their constants, you don't race two models. The integrity score — it's graph + rules, not a trained classifier (training a "collusion detector" on synthetic labels would be circular). And the router decision — frozen policy by mandate; a challenger black box there would break auditability and the "human decides" guardrail.

Promotion rule (one line): a challenger replaces a champion only when it wins on the eval lab's labeled metric and remains explainable. Accuracy alone never promotes an uninterpretable model — interpretability is a gate, not a tiebreaker.

Capstone-honest scope: for the demo you ship all champions (deterministic). The challengers are the "ML in perception" story you turn on for scores 1–3 in Days 3–4, with the eval panel showing champion-vs-challenger side by side. You are never required to train a model to demo — but Classification, Fit, and Ppred are where doing so visibly adds value.
§ ref

Model Glossary — what each method actually is

~6 min

Plain-language descriptions of every method named above, so the inventory reads clearly whether or not you live in ML. Each says what it is, what we use it for, and how interpretable it is — because interpretability is a promotion gate here, not a nice-to-have.

▸ Trained ML methods ML

Champion baseline

Threshold rules

What it is: explicit if/else cutoffs on raw features — registered_days_ago ≤ 14 → new. Not "trained," but it fills the ML slot as the day-one baseline.
Used for: the Classification champion; the fallback for any perception score.
Interpretable: total — you can read the rule.

Challenger

Multinomial logistic regression

What it is: a linear model that outputs a probability for each class and picks the highest; every feature carries a readable weight (coefficient).
Used for: the Classification challenger.
Interpretable: high — the coefficients are the reason codes ("high vpip pushed toward aggressive_predatory").

Challenger

Gradient-boosted trees (GBM)

What it is: an ensemble of many small decision trees built in sequence, each correcting the last; captures non-linear interactions a straight line misses.
Used for: the Fit session-length regressor; an alternative Classification challenger.
Interpretable: medium — feature-importance / SHAP give reason codes, less directly than logistic.

Challenger

Monotonic GBM

What it is: a GBM constrained so a feature can only move the score one direction — more aggressive players can only raise predation pressure, never lower it.
Used for: the Ppred challenger.
Interpretable: medium-high — the monotonic constraints make it predictable and defensible.

▸ Heuristic / rule methods RULE

Champion baseline

Compatibility matrix

What it is: a hand-built lookup table mapping archetype × table style/stakes → a fit score.
Used for: the Fit champion.
Interpretable: total — it's a table you can read and edit by hand.

Verdict rule

Convergence-rule aggregation

What it is: counts how many independent evidence families (device, co-seating, soft-play, timing) point at the same group, subtracts counter-evidence, and maps the total to a band.
Used for: the integrity verdict (low/neutral/high/manual_review).
Interpretable: total — "3 families converged, no counter-evidence → high."

▸ Graph methods GRAPH

Find the ring

WCC — Weakly Connected Components

What it is: a graph algorithm that finds blobs of nodes reachable from one another; each connected blob is a candidate ring boundary.
Used for: drawing a cluster's outer boundary before it's scored.
Interpretable: structural — it's literally "who is linked to whom."

Split the ring

Louvain community detection

What it is: splits a large connected component into tightly-knit sub-communities by maximizing internal edge weight.
Used for: separating a genuine tight ring from a crowd that merely shares a popular IP/device.
Interpretable: structural — communities map to readable sub-groups.

Stretch only

Graph Neural Network (GNN)

What it is: a neural net that learns a vector "embedding" for each node from its graph neighborhood, to spot rings the rules miss.
Used for: integrity scoring — stretch goal, not the demo path.
Interpretable: low — which is exactly why it stays a stretch, never the live verdict.

▸ Deterministic & policy methods DET POLICY

Formula

Deterministic formula

What it is: explicit arithmetic over fields — ratios, rolling-window averages, clamps. No learning.
Used for: Pfrag, Pbleed, the Health(T) composite, ΔHealth.
Interpretable: total — you can trace every number by hand.

The decision

Frozen weighted policy

What it is: a fixed linear formula (w·Fit + w·Health + w·ΔHealth) whose weights are tuned offline but whose form never changes and is never learned end-to-end.
Used for: the router decision + integrity hard-gate.
Interpretable: total — and deliberately so, for audit and human override.

The pattern across the glossary: every method that touches the decision is maximally interpretable (rules, formulas, frozen policy). The lower-interpretability methods (GBM, GNN) live only in perception, behind the eval lab's promotion gate — and the least interpretable one (GNN) is the only method we keep as a stretch rather than a demo path. Interpretability falls as you move away from the decision, never toward it.
§04

Classification — 9 archetypes

~4 min
01Player Classificationfeatures → archetype ∈ {9}

Assigns each player one of 9 archetypes. It is the root input — Fit, Health composition, and the integrity lens all consume it. The archetype is not in the data; P3 computes it from behavioral features.

● Champion
Threshold rules
Straight from players.json's inference note: registered_days_ago≤14 → new, vpip≥0.54 & pfr≥0.40 → aggressive_predatory, promo_redemptions_30d≥8 → promo_hunter, cluster_id set → cluster_member, etc. Deterministic, demo-ready Day 2.
● Challenger
Multinomial logistic / GBM
Trained on the same features; coefficients (logistic) or split-importance (GBM) become the reason codes. Handles overlap the thresholds miss (a loose-passive regular vs. a recreational). Interpretable → eligible for promotion.
Inputs
vpip · pfr · aggression_factor · registered_days_ago · lifetime_hands · avg_session_minutes · sessions_last_30d · promo_redemptions_30d · bot_similarity_score · timing_regularity + presence of cluster_id/household_id/device_group_id
Output
1 of: new · recreational · regular · grinder · aggressive_predatory · promo_hunter · cluster_member · shared_device_household · healthy_anchor (+ confidence)
Depends on
Nothing — root score. Build this first.
Demo test

How well a player suits a table — predicted session length and enjoyment given P's archetype and T's style/stakes/composition. First term of the router.

§06

Health(T) and its four terms

~6 min

Health(T) = 100 − P_pred − P_frag − P_clus − P_bleed. The composite is a deterministic aggregation — not a model. The interesting model choices live in the terms: one is a candidate for ML, two are pure formulas, one comes from the graph.

03Ppred — Predation Pressure0–45
● Champion
Skill-weighted ratio formula
aggressive_predatory weight ÷ recreational cohort, through a pressure curve. Interpretable, the D0b-default.
● Challenger
Monotonic GBM
Regress recreational retention on composition; monotonic constraints keep it auditable. Promote if it predicts churn better on sim outcomes.
Depends on
Classification (composition archetypes)
04Pfrag — Fragility0–25DET

Pure formula: occupied/max ratio + paid_seat_time_trend. No model, no challenger — calibrate the constants against sim outcomes. Inputs all present in table_roster.json.

05Pbleed — Observed Bleed0–20DET

Rolling-window formula over the last N paid-seat-time intervals — recreational truncations vs. archetype baseline. The observed, trailing term; held fixed between seatings. Calibrate window length + threshold; no competing model.

Pclus isn't built here — it's the integrity score (§07) mapped to a 0–30 penalty. That's the one cross-layer dependency inside Health: the graph feeds the table score.
§07

Integrity Score — graph, not a classifier

~4 min
06Integrity Score→ low · neutral · high · manual_review

Per-cluster risk band from convergence of independent signal families, net of counter-evidence. Deliberately not a trained model — training a collusion classifier on synthetic labels would be circular and dishonest.

● Champion
WCC/Louvain + convergence rules
Community detection finds candidate rings; a rule counts how many independent evidence families (device_link, co_seating, soft_play, timing) converge, weighted by confidence, minus counter-evidence. Stated thresholds → auditable.
● Challenger (stretch)
GNN cluster embeddings. A graph neural net over the signal edges — purely a stretch goal. The demo does not depend on it; rules are sufficient and the GNN needs real graph data to be honest.
Inputs
relationships.json: clusters, soft_play_delta (≤−0.60 fires), co_seating{rate}, device_link, timing_correlation, counter-evidence
Depends on
Graph construction from P2 signals (no classification dependency)
Demo test
CL-001 → high (4 families converge) → hold · household → neutral → monitor · CL-002 → neutral (sub-threshold)
Guardrail baked into the method: counter-evidence (household_counter_evidence, low_sample_size_counter_evidence) holds a case at neutral/monitor — "no single signal is proof" is the convergence rule itself, not an afterthought.
§08

ΔHealth & Seating Risk — derived, no new models

~3 min
Deterministic

ΔHealth(P→T)

Re-score Health's composition terms with P added, take the delta. A function of Health(T), not a separate model. O(1). Depends on Health terms + Classification.

Deterministic

Seating-risk score

Composite + reason codes: Health band + ΔHealth + new-player vulnerability + integrity gate. Drives "new-player risk = HIGH." Depends on Health, ΔHealth, Classification, Integrity.

§09

Router — the one place with no model

~3 min
08Rank(P,T) → lobby badgewfit·Fit + wh·Health + wΔ·ΔHealth

The decision. A frozen weighted policy with an integrity hard-gate first. The weights are calibrated offline (grid/Bayesian search on the eval lab) but the form is fixed and human-designed.

No challenger model — on purpose. An end-to-end routing model would be a black box: un-auditable, un-explainable, un-overridable. That breaks the regulator story and the "system recommends, human decides" guardrail. We tune weights; we never swap in a learned decision.
Depends on
Fit (5), Health (6), ΔHealth (8), Integrity gate (7)
Demo test
P-104 → T-8 Recommended (55.2) · T-14 Good fit (44.2) · T-22 Available (13.0)

§11

Build Order

~2 min
  • 1 · Classification (champion) — threshold rules. Unblocks everything. Day 2.
  • 2 · Integrity score (champion) — graph + convergence rules. Independent of classification, so it can be built in parallel. Day 2–3.
  • 3 · Health terms — Pfrag/Pbleed formulas + Ppred champion; wire Pclus from the integrity score. Then compose Health(T). Day 3.
  • 4 · Fit champion + ΔHealth + Seating risk — once classification and health exist. Day 3–4.
  • 5 · Router — assemble Rank, calibrate weights on the eval lab, emit badges. Day 4.
  • 6 · Challengers (1–3) — train the interpretable models, race them in the eval panel, promote winners. The "ML in perception" reveal. Day 4–5.
  • 7 · Evidence packet — P3 assembles, P4's LLM explains. Parallel from Day 3, needs the score outputs.
The whole demo runs on champions. Challengers are additive proof of the ML story, gated by the eval lab — never on the critical path to a working demo.

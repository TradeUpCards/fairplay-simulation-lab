1.The core demo — remaining critical path must-do

This is the PRD demo and the actual objective. The spine:

lobby recommendation  →  pit-boss review/override  →  Standard-vs-FairPlay 8h sim  →  eval evidence

The left two-thirds (lobby, pit-boss, scoring) is built and computed. The gap is the right end: the AI Investigator (P4) — the project's third pillar — does not exist yet (zero LLM code in the repo). Everything below is P4-shaped and unblocks the demo's headline moment. (Full status: docs/learn/demo-readiness.html.)

Step 1 — Evidence Packet (Contract 3) EFFORT: M

The structured object the AI is allowed to see — the seam that enforces "the LLM is never the detector."

Reuses: P3 already produces every input — scores, reason codes, and counter-evidence live in backend/scoring/{health,integrity,seating,classify,router}.py → data/derived/*.json. New: a small assembler that emits one packet per case.

  • Lock the schema (resolves open decision D3): case_id, case_type, scores, top_evidence, counter_evidence, uncertainties, recommended_action, allowed_actions — structured fields only, no raw player rows.
  • Write the assembler; emit packets for the 3 mandatory cases (A new-player, C cluster, E shared-device) first, then F/G.
  • Unlocks: something for the AI to consume; and a demo talking point — "the AI never touches raw data; here is the exact, bounded evidence it sees."

Step 2 — AI Investigator (P4) EFFORT: M

The headline demo moment: the operator opens a case and the AI writes the case — grounded, hedged, counter-evidence surfaced, recommending a human action, never accusing.

Reuses: the evidence packet (Step 1); the latest Claude model (check docs/.../claude-api reference for the current id — don't hardcode). New: the guardrail prompt + output checks + UI wiring.

  • Guardrail system prompt: only use packet evidence · surface counter-evidence · uncertainty language · distinguish health-risk from integrity-risk · always recommend a human action · never state a player cheated.
  • One call: packet in → safe summary out. Add hard-rule output validators (reject/flag any accusation or enforcement language).
  • Wire the summary into the existing pit-boss case view (alongside the structured scorecard).
  • Unlocks: the third pillar made visible; answers the skeptic's "where's the AI?"; demonstrates the guardrails live.

Step 3 — Eval harness EFFORT: S–M

Grades the AI's summaries so "it's trustworthy" is shown, not asserted.

Reuses: the 7 seeded cases (data/seeded_case_labels.json) and the existing EvalPanel.tsx UI shell. New: a rubric + runner.

  • Rubric: grounded (no facts outside the packet) · no over-claiming · counter-evidence present · always a human action.
  • Run over all 7 cases; render pass/fail in the EvalPanel (today only A/C/E have real pass/fail).
  • Unlocks: the PRD's "AI rubric" eval row that is currently "not yet."

Step 4 — (optional) Promote playsim into the sim paths EFFORT: S

Replace the hand-authored data/room_metrics_*.json with computed simulator output so the 8h divergence is real, not presenter-authored.

Reuses: playsim/playsim/room_export.py::derive_room_metrics() — the schema already matches (a test pins it). New: a promotion script + swap. Do this only after Steps 1–3.

Core-path summary
Steps 1→2→3 are the whole gap to a thesis-complete demo, and Step 1 is the unlock. None of them require more simulator depth. This is the work that turns "a health dashboard" into "an AI copilot that explains its reasoning safely."

2.Optional Track A — DES room-economics layer additive · strengthens FairPlay economics

What it is (Sargon's Idea 1): a fast discrete-event simulation above playsim that models the room as events (arrival, seat, decline, leave, re-seat, break, wait-timeout) instead of dealing every hand. It runs thousands of room-days quickly and produces confidence intervals and sensitivity sweeps the current hand-by-hand sim can't (it does ~3 min/run, a few seeds, means not CIs).

What it adds to the demo: turns the Standard-vs-FairPlay beat from "means over 3 seeds" into an investor-grade decision surface — "across 10,000 room-days, here's the paid-seat-time distribution by policy, and here's what would have to be true for FairPlay to win."

MVP stepDetail
Reuse the room loop & behavioral paramsDES = the shipped room loop minus hand-dealing. Transition rates derive from the same calibrated PlayerBehaviorPolicy — single source of truth, so the two models can't quietly contradict.
New: playsim/playsim/des.pySeeded event queue + a SeatEconomicsModel (accept / churn / re-seat / wait-balk / table-survival probabilities). Standard, Random, FairPlay, most-full policies.
New: analysis harnessRun 1,000+ room-days; emit summary JSON with CIs + sensitivity sweeps over arrival rate and liquidity.
Honest caveat (Sargon's, and correct)
Scale is not validity. DES inherits playsim's hand-authored leave model, so a "FairPlay wins/loses" result from playsim-derived priors is the same baked-in story at scale, not independent evidence. Lead with sensitivity ranges ("what would have to be true"), treat priors as coverage, and reserve verdicts for real operator data. Frame outputs as illustrative until calibrated.

Effort: SMALL — days — it reuses the room loop and params; the only new work is the event queue and the analysis harness.

3.Optional Track B — Training-simulator POC additive · AI-product demo

Your vision: a human plays a realistic 6-max table against configurable bots, as a POC (not hardened production), to: (1) learn to play better against specific player types, (2) identify player types when they're not known up front, and (3) get AI-assisted post-hand analysis on what they could have done differently and why it's better against that type.

Why this is cheaper than it looks — and tied to the core work
Two of the three pieces already exist in playsim/, and the third reuses the exact architecture of the core demo's AI Investigator: structured summary → guardrailed LLM → explanation. Capability #3 (the coach) is the AI Investigator pointed at a teaching question instead of a risk question. So building it sharpens the same AI muscle the demo needs. (Per Sargon: the "LLM is never the detector" rule is about risk detection, not teaching — an LLM coach is fine.)

The three capabilities → concrete build

CapabilityReuses (exists)New (to build)

Train vs player types Bot personalities already exist as archetypes in playsim/playsim/agent.py + knobs.py (grinder, calling-station/recreational, aggressive/predator, rock-like reg, solver_like, bot_like). The hand engine is table.py (real NLHE on PokerKit). A human seat + action UI (fold/call/raise), table-state rendering, bot-style picker, fixed blinds/stacks.
2. Strong thematic tie-in to FairPlay. Hide the bot labels; after N hands the human guesses the archetype; reveal + score the guess (optionally show the model's read alongside).
3. AI post-hand coaching (the make-or-break) Per-decision equity from playsim/playsim/equity.py (Chen percentile + Monte-Carlo). Same pattern as the AI Investigator (structured summary → LLM). A hand-summary assembler (the human's decisions + board + opponent archetype/tendencies + equity at each decision) → a coach LLM call: "you could have done X; vs a calling-station that's better because Y."
De-risk first: coaching quality
Sargon is right that coaching is the make-or-break. "You should have folded" reads thin. Credible coaching needs the structured per-decision context (equity vs the opponent's likely range + the archetype's known leak) feeding the LLM. Build the coach first on a handful of canned hands before any polish on the table UI — if the coaching is good, the POC lands; if it's shallow, no UI saves it.

POC scope (bounded, not production)

  • 6-max table by default (the most common online cash format), one human vs 5 mixed-type bots; 9-handed a config option; heads-up an optional mode. Not HU/3-handed-only — see the note below on why.
  • Start with 3–4 distinct bot styles seated in a realistic mix; fixed blinds/stacks.
  • Hand-history review + the post-hand coach; the blind "guess the type" round.
  • No real-money language, no multiplayer, no accounts, no RL required (the policy seam can take an RL bot later — playsim/playsim/baselines.py already scaffolds an act() adapter).
  • Milestone question: "can a user play 20 hands at a 6-max table of distinct styles and come away understanding what each opponent does differently — with coaching that actually explains why?"
Why 6-max, not heads-up — and where the real scope cost is

Heads-up and short-handed are a different game — much wider ranges, blind-vs-blind dynamics, relentless aggression, position meaning something different. Skills built HU don't transfer to the full table most players actually sit at, and both training goals get richer at a full table: you read a roomful of distinct types (goal 2), and "this table mix rewards patience, not bluffing" (goal 1) is inherently a table-composition idea that barely exists HU.

Crucially, seat count is nearly free on the engine — table.py already deals N-handed NLHE on PokerKit, the room sim already seats up to max_seats, and the bots already exist; seating 5 mixed bots is no more engine work than 1. The one real cost of more seats is multiway coaching (equity/EV across several opponents' ranges is harder than HU). So bound the POC in the coach, not the seat count: v1 coaching focuses on the decisive opponent/spot in a hand ("you were really playing a pot against the station in seat 5"); full multiway-pot analysis is a later coaching-depth phase.

The honest tradeoff (so you choose with eyes open)
Per Sargon, the training game is largely orthogonal to FairPlay's mission — it shares the engine, not the thesis. It advances the capstone / AI-product goal far more than the FairPlay product goal. That's a fine reason to build it — but it is a second product, and its UI + coaching make it the LARGER — week+ track even as a POC. Don't let it crowd out the core path (Section 1) before that's done.

Recommended sequence

  1. Do the core path first (Steps 1–3). It's the refocus, it's the thesis, and it's bounded. Nothing else should start before this is moving.
  2. Then DES (Track A) if the goal is the FairPlay business case — it's a cheap, days-scale additive win that reuses the room loop and makes the economics beat investor-grade.
  3. Then the Training POC (Track B) if the goal is a visible AI-product capstone — bigger, separate, but it reuses the engine + bots and (crucially) the same AI-coach architecture as Step 2, so it compounds rather than diverges. De-risk coaching first.
The one-line decision
Core path is non-negotiable and comes first. After that, DES if you're selling FairPlay economics; the Training POC if you're selling an AI product — and the Training POC is worth it partly because it reuses the AI Investigator you'll have just built. Pick based on which story you're telling; don't run A and B at once without clear ownership.

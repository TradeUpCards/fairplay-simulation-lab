How the playsim engine works

A deterministic, seeded poker simulator: archetype agents play real No-Limit Hold'em hands on the PokerKit engine, and the signals our system consumes — looseness, aggression, soft-play, robotic timing, paid seat-time — emerge from play rather than being asserted. This is the deep reference: the philosophy, what it deals, what it emits, how it parallelizes, how it scores health, why it's reproducible, and exactly where a learned poker brain plugs in.

Where this fits (2026-06): this is the core engine reference. The closed-loop room simulator (Standard-vs-FairPlay routing over a seeded arrival stream, with swappable default, fit-aware, and reason-aware player-behavior models) is built on top of it — for that, see the room simulator guide and the findings in docs/learn/playsim-room-routing-findings.md.

You are here: the engine reference The gentle walkthrough
The engine in one breath

One parameterized policy + ten knob-sets drive agents that play real hands on PokerKit (the referee). We control the deck from a single seeded RNG, so a run is byte-for-byte reproducible. Played hands are aggregated into Contract-1 features (integrity loop) and realized table health → paid seat-time (health/routing loop). Room-scale runs start from the fixed hour-0 fixture room (12 tables, 68 seated players), then route a seeded arrival stream and record exits, breaks, re-seat attempts, and wait-balks. Health is computed independently from chip flow — never by re-running the scoring engine — so the routing backtest isn't a tautology. A learned brain (RLCard / OpenSpiel / PokerRL) drops in behind one act() method, with PokerKit still refereeing.

Two loops, one engine

Integrity loop — run

Does the detector catch bad behavior? Stacks reset each hand for stable feature calibration; soft-play and the bot's flat timing emerge from play.

Health/routing loop — routing

Does routing retain more paid seat-time? Stacks persist, players log off when tilted, and the same cohort is compared across two table compositions under matched seeds.

Engine modes: population, run_session, and room-sim

The engine can be entered three ways. The first two are the original hand/session engines; the third is the closed-loop room simulator that now handles dynamic arrivals, routing, table breakage, and re-seat behavior.

Population quota-leave

population_run.run_population loads the fixture tables, derives a stable per-table seed, passes each player's target hands into run_session, and merges hands/features afterward. A player leaves after reaching min(lifetime_hands, cap), which keeps low-volume fixture players from being over-dealt.

Free-form sessions

run_session initializes agents, controls the deck from a seeded RNG, rotates the dealer, calls play_hand, updates stacks and seat minutes, records leave/bust events, and aggregates the result. It is the reusable table loop under both fixture and room-scale experiments.

# Fixture-backed population corpus
python -m playsim.cli population --hands-cap 60 --seed 42

# Closed-loop room routing comparison
python -m playsim.cli room-sim --policy fairplay --seed 42 --behavior reason-aware

# Free-form session from Python
from playsim.runner import Player, run_session

players = [
    Player(player_id=1, archetype="recreational"),
    Player(player_id=2, archetype="grinder"),
]
result = run_session(players, n_hands=100, seed=42)
Important boundary: population mode is a deterministic hand-corpus and feature-production path, not a room-routing test. Dynamic seating and re-seating live in room-sim, where the policy sees current table occupancy and open-seat state after each arrival or leave event.

BPhilosophy

Four convictions shape every design choice. They're worth internalizing because they explain the things that look unusual (why we control the deck, why health is computed twice, why there's a "skill-edge" fudge).

1 · Signals must emerge from play — never be assertedcore▶

The static generator writes each player's vpip, soft_play_delta, timing_regularity. That makes the detector grade its own homework — it just reads back a field it was handed. playsim instead has an agent play hands in character and measures the stats from how it played. A colluder actually checks down to a teammate; a bot actually has flat action timing. If a signal can be recomputed from the played hands, it's a genuine test of the detector.

2 · The engine is the referee; the agent only decidesPokerKit▶

PokerKit (UofT Computer Poker Research Group, MIT, the same group behind the PHH format) owns legality, betting rounds, side pots, showdown evaluation, and chip distribution. An agent can never make an illegal move — it proposes an action and the engine validates and applies it. This separation is what lets us swap the agent's "brain" (heuristic today, a learned solver tomorrow) without touching game logic, and it's why the simulation is correct poker rather than an approximation.

3 · Derive, don't assert — and don't validate against yourselfguardrail▶

The health/routing backtest asks "does routing produce healthier tables?" If the same model that scores health also generates the outcomes, the answer is a tautology. So playsim computes health from first-principles chip flow (who won, lost, busted) — completely independent of the scoring engine's Health(T) formula. See §G; this is the single most important correctness property of the health loop.

4 · Be honest about what's modeled vs. emergentthe skill-edge▶

Heuristic agents (equity thresholds + knobs) do not reliably win chips off weaker players — cards and variance dominate, and a calling station will stack a grinder on a lucky seed. Rather than pretend a skill edge emerges when it doesn't, the engine applies a small, transparent, zero-sum per-hand skill-EV transfer (skill_edge) from weaker to stronger contestants — a real, measurable bb/100 quantity. It's a behavioral input; health and seat-time stay derived from the resulting chip flow. A trained brain (§H) would make the edge genuinely emergent and the transfer unnecessary. We document it rather than hide it.

CDoes it deal hands, or hand outcomes?

It deals real hands, street by street — then derives outcomes from them. This is the difference between a genuine simulation and a statistics generator.

✗ What it is NOT

It does not sample an outcome ("P-104 loses 40bb this hand") from a distribution. There is no shortcut that skips the play.

✓ What it actually does

Deals hole cards, runs preflop → flop → turn → river betting with real positions/stacks/pot odds, deals the board, resolves showdown via PokerKit's evaluator, distributes chips.

On every decision the agent (1) estimates equity — preflop via a Chen-style percentile over all 1326 combos, postflop via Monte-Carlo vs. random opponent ranges using PokerKit's hand evaluator; (2) lets its knobs bias the thresholds (looseness lowers the equity needed to continue, aggression turns a marginal call into a raise); (3) picks an action and a size. The hand is a real object with a full action log, board, and per-player payoffs.

The one nuance: a post-hand outcome adjustmenthonest▶

After a hand resolves, the skill-edge transfer (§B-4) adjusts the payoffs by a small zero-sum amount based on the skill gap among the players who contested the pot. So the play is fully dealt and refereed, but the realized chip result carries a modeled skill correction on top. This is the only place an "outcome" is touched directly — and it's deliberately small, zero-sum, and documented. Set skill_edge=0 for pure-emergent play (then the integrity signals are untouched; only the health-loop economics lose their reliable edge).

# table.py — one decision, refereed by PokerKit
obs = Observation(street, hole, board, to_call, pot, min_raise_to, max_raise_to, ...)
d   = agent.act(obs, rng)                 # fold / check_call / raise(amount)
if d.kind == "raise" and state.can_complete_bet_or_raise_to(d.amount):
    state.complete_bet_or_raise_to(d.amount)   # engine enforces legality
elif d.kind == "fold" and state.can_fold():
    state.fold()
else:
    state.check_or_call()

EDeterminism — exactly how

Same (roster, seed, params) → byte-identical result, every time. This is a hard rule (the demo must replay from frozen fixtures), and it's achieved by routing all randomness through one seeded source.

  • One RNG seeds everything. A single random.Random(seed) drives the deck shuffle, every agent's mixed-strategy coin-flips and bet-sizing jitter, the Monte-Carlo equity sampling, and the leave-model session jitter. Nothing reads the global random or wall-clock time.
  • We control the deck — we don't trust PokerKit's shuffle. Dealing automations are turned off; we shuffle our own 52-card deck with the seeded RNG and deal each card explicitly via state.deal_hole(c) / state.deal_board(c). Determinism is by construction, not by hoping the library's internal RNG is seeded.
  • The button rotates by remapping players to seats (seat = (dealer + s) % n), so positions rotate without touching PokerKit's blind posting — and the rotation is a pure function of the hand index.
  • The replay command asserts it — it runs the same seed twice and checks the features and every hand's payoffs/board are identical.
# table.py — deterministic dealing from our own seeded deck
rng = random.Random(seed)
deck = FULL_DECK[:]; rng.shuffle(deck); it = iter(deck)
# dealing automations OFF → we supply every card
for _ in range(2):
    for seat in range(n):
        state.deal_hole(next(it))            # hole cards
while state.can_deal_board():
    for _ in range(state.board_dealing_count):
        state.deal_board(next(it))           # flop=3, turn=1, river=1
Freeze-and-replay. Because a run is a pure function of its inputs, the expensive work happens once, offline; the output is committed as JSON and the demo replays the frozen file. The seed is the artifact. This is the same pattern the rest of the lab uses for its fixtures.

FParallelization scheme

Be precise here: a single session is sequential and deterministic by design. The parallelism lives one level up — across seeds — and the determinism is exactly what makes it embarrassingly parallel.

Why a single hand can't parallelize

Poker is inherently sequential within a hand: each action depends on the prior one (you can't decide the turn before the flop is dealt and acted on), and each agent's RNG draws must happen in a fixed order or determinism breaks. So play_hand and the hand loop in run_session are deliberately single-threaded.

Where it does parallelize — the Monte-Carlo over seeds

The health/routing result averages many independent seeded runs (individual fish run hot/cold; the population mean is the signal). Each (roster, seed) pair is a pure, independent function — no shared state, no ordering constraint between runs. That's the textbook embarrassingly-parallel shape.

Current state (honest): the seed loop in service.simulate_routing() runs sequentially today — correctness and reproducibility first. Because each run is independent and deterministic, dropping it onto a multiprocessing.Pool (or a job queue / Ray / Spark) over seeds is a near-trivial change with no determinism risk: worker i gets seed i, results are reduced by averaging. The Monte-Carlo equity inside an agent is likewise per-decision and could be vectorized, but it's intentionally cheap (~12–30 samples) instead.

# the parallelization seam (today: a sequential loop; trivially a Pool.map)
runs = [compare_routing(
            run_session(std_roster, hands, seed=s, ...),
            run_session(fp_roster,  hands, seed=s, ...))
        for s in range(seed, seed + seeds)]      # <- map over workers
headline = average(runs)                              # <- reduce

GHow it calculates table health

The key question, answered plainly: playsim does not call the canonical scoring/ engine, and it does not copy its Health(T) formula. It computes a separate, independent health measurement from raw chip flow — on purpose.

why independence is mandatory
If playsim computed health by re-running the scoring engine's model, the routing backtest would be circular — the router would "succeed" by construction because the outcome was generated from its own assumptions. Independence is the whole point: realized health (bottom-up, from chips) is a genuine, external check on the scoring engine's predicted Health(T) (top-down).

What playsim's health.py measures (first-principles)

  • Recreational loss velocity — bb bled per 100 cohort hands actually played (normalized by cohort hands, not session hands, so a table where the cohort quits fast isn't wrongly flattered).
  • Winnings concentration — the top winner's share of all winnings (predation concentration).
  • Beginner bust rate — rec/new rebuys per 100 cohort hands.
  • Predation — net bb the skilled cohort extracted per 100 session hands.

These combine into a transparent 0–100 health_score (bleed-dominant) and a band (healthy / fragile / beginner_unfriendly). For the north-star health/routing loop they feed the real outcome: retained paid seat-time — players log off when their tilt-shortened session budget runs out, so a faster bleed → shorter sessions → less seat-time.

So the relationship is: the canonical scoring/ engine remains the source of truth for the scores the product ships; playsim is the independent world those scores are validated against. They share no health code — by design.

HWhere RLCard / OpenSpiel / PokerRL plug in

The whole engine is built around one swappable seam: the agent brain. Anything that can map a game state to an action — a heuristic, a CFR solver, a trained net — slots in behind the same interface, with PokerKit still refereeing legality.

The seam

# every brain implements exactly this:
class Brain:
    def act(self, obs: Observation, rng) -> Decision: ...

# Observation: street, hole, board, to_call, pot, min/max_raise_to,
#              my_stack, n_active, live_opponent_ids, ...
# Decision:    fold | check_call | raise(amount)   (+ flags for stats)

The heuristic ArchetypeAgent implements it today. A learned brain is just a different implementation; playsim/baselines.py already contains the RLCardAgent adapter scaffold and the immediately-usable built-in solver_like archetype.

What stays the same, and what gets betterdesign payoff▶

Unchanged: PokerKit still refereeing (legal moves, showdown, chips); the seeded deck and determinism; the feature aggregation; the health/seat-time metrics; the output formats. You only replace what's inside act().

Better: a genuine solver brain plays a real, consistent chip-EV edge — so the skill_edge transfer (§B-4) becomes unnecessary and the predation→decay→seat-time signal becomes fully emergent instead of modeled. That's the principled end-state; the heuristic + skill- edge model is the pragmatic V1 that ships today.

One caveat: an external brain that uses its own RNG or GPU non-determinism would break reproducibility. The adapter must either be deterministic given our seed, or its outputs must be generated once and frozen (§E) — the same escape hatch the rest of the lab uses for any non-deterministic source.

FairPlay Simulation Lab

Two poker simulators, different jobs.

The Superpowers/backend simulator proves that archetype behavior can emerge from legal poker hands. playsim takes that substrate and turns it into a routing validation lab: retained paid seat-time, independent realized health, and eventually a router-in-the-loop backtest.

backend/sim: feature generator playsim V1: fixed counterfactual backtest playsim V3: router-in-the-loop target
00

Summary

Both systems use agentic poker play, but they answer different questions. The backend simulator asks "can we generate credible player behavior stats from played hands?" playsim asks "if FairPlay changes seating, do realized table outcomes improve?"

The short version

backend/sim is a standalone outcome generator. It deals legal No-Limit Hold'em hands, lets archetype agents act, and rolls the action log into vpip, pfr, aggression_factor, avg_pot_size_bb, net_chips, and hand histories.

playsim is a validation harness. It still plays hands, but it adds FairPlay-specific integrity behaviors, retention, paid seat-time, realized health, Standard-vs-FairPlay comparison, multi-seed averaging, and a service boundary for UI/API use.

The caution

The current playsim V1 backtest uses fixed routing_standard and routing_fairplay rosters. That is useful, but it tests the routing hypothesis, not the actual router implementation.

The stronger backtest is V3: the simulator starts from a room snapshot, calls the canonical router, seats the cohort where the router recommends, then measures realized outcomes independently.

02

What they share

The systems are not opposing designs. playsim is best understood as a FairPlay-specific fork/evolution of the same core insight: use a real poker engine and seeded archetype agents so behavior is measured, not typed in by hand.

Engine seam

Both rely on PokerKit for poker mechanics and keep the rest of the stack engine-agnostic. The agents decide; the engine enforces legality, betting, pots, board state, and payoffs.

Seeded agents

Both reject live LLM hand execution. Agent decisions are seeded and replayable, with archetype knobs controlling style and strength.

Derived features

Both treat played hands as the source for behavior stats such as VPIP, PFR, aggression, pot size, and profitability. That avoids "the fixture says the stat because we typed it."

Foundation Superpowers/backend sim playsim
Poker mechanics PokerKit adapter behind Engine. PokerKit referee in table.py, with a higher-level play_hand record.
Agent abstraction Agent.decide(ctx, rng) over DecisionContext. ArchetypeAgent.act(obs, rng) over Observation.
Determinism Master seed plus derived dealing/decision RNG streams. Seeded deck and agent decisions; replay command asserts byte-identical output.
Primary output data/sim/hand_histories.json and data/sim/player_stats.json. out/sim.db, PHH-shaped hands, features.json, service JSON, and routing comparison metrics.
03

Superpowers/backend sim: the feature generator

The design in docs/superpowers produced backend/sim. Its purpose is narrow and valuable: generate legal poker outcomes and roll them into player behavior features.

Design center

A static config describes tables, seats, archetypes, stack size, blinds, and hands per table. For each hand, the driver resets stacks, rotates the button, deals a seeded deck, asks each agent for an action, and logs action/result rows. 

config -> run_table -> EventLog -> stats.rollup
       -> data/sim/hand_histories.json
       -> data/sim/player_stats.json
Agent model

The backend simulator has seven behavioral archetypes. Each archetype is a skill percentage plus style knobs: aggression, tightness, and bluff frequency.

  • Skill blend: with probability skill, use the strong policy; otherwise use beginner policy.
  • Preflop range gate: Chen-style hand strength plus position and tightness.
  • Style overlay: aggression upgrades calls, bluff frequency turns some folds into tagged bluffs.

What its scoring means

vpip = preflop voluntary-money-in-pot hands / hands dealt
pfr = preflop-raise hands / hands dealt
aggression_factor = postflop bets-and-raises / postflop calls
avg_pot_size_bb = mean pot size on contested hands
net_chips = sum of per-hand payoffs
Important limitation

The backend simulator explicitly defers seat-time. It reloads stacks between hands, keeps seating static, and does not model join/leave/quit. That is correct for a feature generator, but it means it cannot by itself answer the product's north-star question: did routing retain paid play-time?

04

playsim V1: the health/routing validation sandbox

playsim keeps the core gameplay idea, then adds FairPlay-specific behaviors and outcome metrics: cluster soft-play, target-weak predation, bot-like timing, retention, paid seat-time, and independent realized health.

Design center

Instead of only asking whether features emerge, playsim asks whether a seating choice changes the lived experience of vulnerable players. It can run one table for integrity signals or compare Standard vs FairPlay routing across many seeds. 

roster -> run_session(retention=True)
       -> played hands
       -> features
       -> realized health
       -> paid seat-time retained
Agent model

playsim expands the archetype vocabulary to match FairPlay: cluster_member, shared_device_household, bot_like, and solver_like join the regular poker personas.

  • Looseness and PFR: tuned to hit fixture medians.
  • Postflop aggression: calibrated by calibrate.py.
  • Soft play: cluster members reduce aggression against teammates.
  • Target weak: predators and clusters increase pressure against weak opponents.
  • Timing regularity: bot-like agents produce low-jitter latency.

What playsim measures that backend/sim does not

Metric Meaning Why it matters
paid_seat_hours_cohort How long the routed vulnerable cohort actually stayed seated. The north star: healthier routing should slow play decay.
avg_casual_session_min Mean session minutes for the protected cohort. Demo-friendly evidence: "12 min vs 42 min" is legible.
retention_rate / early_exits Whether cohort members remain active or leave early. Separates healthy volume from churn-throughput.
recreational_loss_velocity BB lost per 100 cohort hands. Connects chip flow to session decay.
winnings_concentration Share of all winnings held by the top winner. Captures predatory concentration that averages can hide.
The V1 tradeoff

playsim V1 currently compares fixed routing_standard and routing_fairplay rosters. This is a controlled counterfactual: same cohort, matched seeds, different table composition. It proves the hypothesis that good routing can improve outcomes. It does not yet prove the canonical router would choose the good table.

05

V1, V2, V3: what changes

The V1/V2/V3 framing is a roadmap for increasing proof strength. Each step makes the simulator more realistic or more integrated, but also increases complexity and coupling.

Version Question answered How it works What it proves What it cannot prove
backend/sim Can gameplay generate behavioral stats? Static tables, reset stacks, hand histories, per-player rollups. Archetype behavior can emerge from played hands. Routing impact, seat-time retention, live router correctness.
playsim V1 Does better table composition retain more play-time? Fixed Standard vs FairPlay rosters, same cohort, matched seeds, retention mode. The routing hypothesis is directionally sound. The actual router would pick the right table.
playsim V2 Do stronger poker brains change the results? RLCard/OpenSpiel/PokerRL or exported solver-like policies behind the same act() seam. The result is less dependent on heuristic-agent quirks. Full room dynamics unless routing and join/leave are also integrated.
playsim V3 Does the actual FairPlay router improve outcomes? Initial room snapshot, Standard baseline chooser, FairPlay router call, playsim outcome evaluation. Router implementation quality, not just routing theory. Production accuracy without real deployment data and longitudinal calibration.

The V3 loop

1
Initial room Same players, tables, open seats, and seeds.
2A
Standard choice Naive lobby, historical route, or random open table.
2B

3

06

Technical nuances in scoring

The word "health" appears in multiple places. That is fine only if the page is explicit about which score is predicted, which score is realized, and which metrics are just behavior rollups.

Canonical scoring

Product score

The scoring engine in backend/scoring computes the product's predicted table health:

Health(T) = 100 - P_pred - P_frag - P_clus - P_bleed

This is composition-led. It uses archetype classifications, occupancy/trend, seated cluster severity, and observed recreational truncation.

Router score

Decision score

The router ranks candidate tables with a frozen auditable formula:

Rank = 0.30*Fit + 0.40*Health + 0.30*DeltaHealth

This is what V3 should call before gameplay. It predicts the best table; it should not be used as the answer key after gameplay.

Realized health

Backtest score

playsim computes a separate realized health measurement from chip flow:

100 - bleed penalty - concentration penalty - bust penalty

This is intentionally not the canonical Health(T) formula. It is an independent outcome measure used to test the prediction.

Why this matters

If the router chooses a table using Health(T), and the backtest declares victory by re-running the same Health(T), the result is circular. The correct setup is: predicted health before routing, independent realized health after gameplay.

07

Is backtesting done the same way?

No. The backend simulator has tests and calibration runs. playsim has a backtest. The live backend room API has mutable rescoring, which is useful for interaction but is not itself an outcome backtest.

System What "test" means Backtest status Correct interpretation
backend/sim Run deterministic tables and verify rollups/calibration. Not a routing backtest. Useful for generating emergent behavior features and validating simulator determinism.
backend/app live scoring Stand/sit players, recompute affected table health, stream updates. What-if rescoring, not outcome simulation. Useful for the pit-boss UI and live demonstration of recomputation.
playsim V1 Fixed Standard roster vs fixed FairPlay roster, averaged across seeds. A real counterfactual, but fixed-path. Tests whether healthier composition produces better realized outcomes.
playsim V3 Call router, play out its chosen table, compare against baseline choice. Target design. Tests whether the actual router improves retained paid seat-time.

The backtest recipe

for each seed:
  room0 = same initial room snapshot

  standard_table = baseline_policy(room0, arriving_player)
  fairplay_table = router.route(arriving_player, room0).top_recommendation

  standard_outcome = playsim.play(room0 + standard_table, seed)
  fairplay_outcome = playsim.play(room0 + fairplay_table, seed)

  compare:
    paid_seat_hours_cohort
    avg_casual_session_min
    retention_rate
    early_exits
    realized_health
    rec_loss_velocity
08

What we gain and lose

backend/sim

feature generator

Gain:

  • Small surface area and easy determinism.
  • Clean hand-history and player-stat outputs.
  • Good unit-test boundaries: deck, tools, policy, engine, stats.
  • Useful provenance for behavior features.

Lose:

  • No retained seat-time, leave model, table breaks, or paid-hour north star.
  • No FairPlay-specific cluster/household/bot behavior in v1.
  • No router integration and no Standard-vs-FairPlay outcome comparison.
  • Reload-to-stack means no true session decay.

playsim

validation lab

Gain:

  • FairPlay archetypes map directly to the PRD.
  • Integrity signals can emerge from play: soft-play, target-weak, robotic timing.
  • Retention mode connects chip flow to paid seat-time.
  • Multi-seed routing comparison reduces poker variance.
  • Service API is closer to UI/API consumption.

Lose:

  • More moving parts and more assumptions.
  • Skill-edge transfer is a modeling input until stronger brains make it emerge.
  • V1 fixed rosters can be oversold if not labeled carefully.
  • V3 router-in-the-loop will add coupling to canonical scoring contracts.
09

Decision guide

Use both, but do not ask them to prove the same thing. The cleanest story is a ladder: backend/sim proves emergent behavior; playsim V1 proves the table-health hypothesis; playsim V3 proves router implementation quality.

Use backend/sim when
  • You need raw hand histories.
  • You need player behavior rollups.
  • You are testing policy realism or determinism.
  • You want low-coupling data generation.
Use playsim V1 when
  • You need a demo-safe Standard-vs-FairPlay comparison.
  • You need paid seat-time as the north-star result.
  • You want to show why healthier composition matters.
  • You can honestly say "fixed counterfactual, not router proof."
Build playsim V3 when
  • You need to prove the actual router selects better tables.
  • P3's router contract is stable enough to call.
  • You can define a fair Standard baseline chooser.
  • You can freeze outputs for the demo after generating them.

Recommended narrative: "We started with the Superpowers/backend simulator to make behavioral stats emerge from real hands. playsim turns that into an outcome lab: it measures whether healthier seating retains paid play-time. Today V1 is a fixed counterfactual; the next stronger proof is V3, where the actual FairPlay router chooses the table and playsim independently evaluates the result."

FairPlay Simulation Lab

The playsim engine: fixture population and free-form sessions

playsim is the deterministic poker simulator for FairPlay. It does not assign risk scores directly. It deals synthetic poker hands, lets archetype agents act through a real poker engine, and exports the hand histories and derived features that scoring and model-building can consume.

Update (2026-06): the room-scale router described below as "planned" / "not yet" has since shipped — the closed-loop room-sim compares Standard vs FairPlay routing over a seeded arrival stream. This page documents the underlying engine modes; for the room simulator, its findings, and the behavioral model, see the room simulator guide.

01

Purpose: generate evidence, not conclusions

The simulator exists to produce realistic, inspectable poker evidence. The detector should discover behavior from the output, not receive pre-labeled outcomes from the simulator.

Core idea

Agents play hands

A player is represented by an archetype, such as new, recreational, grinder, bot_like, or cluster_member. The agent chooses poker actions from an observation of the current hand.

Measurement

Features are aggregated

After hands are played, features.aggregate computes VPIP, PFR, aggression factor, timing regularity, soft-play delta, net BB, and related signals from the recorded actions.

Boundary

Scoring stays separate

playsim does not become the risk detector. It creates hand histories and feature rows. The backend scoring layer decides how to classify risk from those signals.

Input Roster

Either fixture tables from players.json and table_roster.json, or custom Player objects.

Policy Archetype agent

Knobs bias looseness, aggression, timing variance, skill, tilt, predator behavior, and soft-play.

Referee PokerKit hand

Legal actions, street progression, showdown, and chip accounting are enforced by the poker engine.

Record HandRecord

Each hand records seats, hole cards, board, actions, payoffs, pot size, and showdown participants.

Output Features and JSON

Exports become training, scoring, QA, and future simulator-panel inputs.

02

Two modes: population and run_session

The new fixture-backed population path did not replace the general simulator. It wraps the same run_session engine with fixture loading, table seeds, export formatting, and hand quotas.

Mode Input source What it is for Routing status
population
current		 data/players.json, data/table_roster.json, and data/derived/classifications.json. Day-2 fixture corpus generation: deterministic, playsim-native hand histories for the seated fixture population, using min(lifetime_hands, cap) as each player's target. Does not yet call the backend router. It replays the hour-0 table roster and uses quota-leave.
run_session
current		 Any caller-provided list of Player(player_id, archetype, cluster_id, household_id). Free-form simulation for calibration, custom experiments, demo rosters, smaller subsets, routing counterfactuals, and direct Python tests. Can run fixed Standard vs FairPlay roster experiments through routing, but not live room-scale routing.
Population mode

Fixture-backed and quota-aware

population_run.run_population loads fixture tables, builds a playsim roster for each table, derives a stable per-table seed, passes quota_hands into run_session, and merges the resulting hands into one corpus.

Important limitation: only players seated in table_roster.json are simulated today.
run_session mode

Still free-form

You can still simulate outside players.json. The core engine accepts any valid roster of Player objects, so tests, one-off experiments, and synthetic demo tables remain useful.

Example uses: case_c, routing_standard, calibration runs, and custom notebooks.
03

Overall approach

The simulator is deliberately modular: fixture loading, agent decisions, poker execution, feature aggregation, health evaluation, and JSON export each live behind a narrow boundary.

Execution loop

What happens inside a session

run_session initializes one agent per player, creates a seeded random generator, rotates the dealer by remapping player order each hand, calls play_hand, and then updates stacks, hand counts, seat minutes, bust counts, and leave decisions. After the loop, it aggregates played hands into a SimResult. 

result = run_session(
    roster,
    n_hands=400,
    seed=42,
    equity_samples=20,
    quota_hands={101: 28, 198: 400}
)
Determinism

Replayability is a product rule

The deck and agent decisions draw from the same seeded RNG. A table seed is derived from the master seed and table id, so fixture runs remain stable table-by-table.

Quota-leave

Why low-volume players stop

Population mode uses quota_hands. When a player reaches hands_played >= target, they are removed from seated after that hand. This fixes the old failure mode where a new player with 28 lifetime hands could be dealt 400 hands just because a grinder at the same table needed 400.

Table break

Why some high-quota players may end early

If quota-leave removes everyone except one player, the table breaks because poker needs at least two players. This is accurate for the current no-backfill fixture mode, but it is also the reason the next phase needs joins and backfill.

04

Variables and why they exist

The simulator separates fixture variables, session variables, archetype behavior variables, and output metrics. This is what keeps the model explainable.

Variable Where it appears Why it matters
player_id Player, fixtures, hand export Stable identity across seats, hands, features, clusters, households, and output indexes.
archetype classifications.json or caller-provided Player Selects the knob vector that shapes looseness, aggression, timing, skill, and session behavior.
cluster_id / household_id Fixtures and Player Preserves relationship context for soft-play and collusion-style feature generation.
lifetime_hands players.json Population mode converts this into a target via min(lifetime_hands, cap).
cap population --cap Bounds the per-player hand count, usually 400, so grinders do not dominate output size.
quota_hands run_session Implements quota-leave. Population passes fixture targets through this general engine hook.
seed / master_seed CLI and runner Makes the simulation replayable. Population derives per-table seeds from the master seed.
equity_samples Agent decisions Controls Monte Carlo equity approximation cost and smoothness for action decisions.
persist_stacks run_session, health mode Lets chip loss and busts accumulate across hands for health-style simulations.
retention run_session, routing demo Turns on log-off behavior for vulnerable cohorts, producing paid-seat-time metrics.
skill_edge run_session Optional zero-sum EV transfer used when heuristic agents need a controlled skill gradient.
hands_per_hour run_session Converts dealt hands into seat minutes for retention and paid-seat-time analysis.
Why this matters: the simulator is not a black box that says "this table is risky." It produces auditable intermediate evidence: action logs, timing, chip flow, hand counts, and derived feature rows.
05

Does it use routing right now?

There are two different meanings of "routing" in the codebase today. The distinction matters.

Current

Fixed-roster routing backtest exists

playsim routing compares predefined Standard and FairPlay rosters under matched seeds. It is a controlled experiment: same vulnerable cohort, different table composition, retention enabled, and realized health measured from chip flow.

works today
Not yet

Population mode does not route arrivals

population currently consumes table_roster.json as the seating source. It does not call backend/scoring/router.py, does not run live join decisions, and does not simulate Standard vs FairPlay room policies over time.

not in population yet
Future routing

What the next routing phase should do

The next phase should convert population mode from a static hour-0 seating replay into a room-scale process: players leave, arrivals seek seats, Standard routing uses baseline table assignment, and FairPlay routing uses predicted table health from the backend scorer. The simulator should then compare room outcomes such as retained paid seat-time, rec loss velocity, table breakage, and health bands.

Guardrail: routing decisions should use predicted health from the scoring stack, not realized health from playsim. Realized health is for after-the-fact evaluation. Using it for routing would create circularity.
06

Next phases in the simulator

The implementation is intentionally staged. Phase A is done; the remaining phases turn static fixture replay into a dynamic room simulator.

done now next
A

Quota-leave in population runs done

Players leave after reaching min(lifetime_hands, cap), preventing low-volume players from being over-dealt.

B

Seat unassigned players next

players.json contains more players than are seated in hour-0 table_roster.json. Backfill or synthetic fill tables are needed for a full population corpus.

C

Export sessions and seat events next

Add session slices and seat events so scoring can reason about bleed, churn, and table tenure rather than only hand-level actions.

D

Health-modulated arrivals and FairPlay routing next

Model seekers arriving over time. Standard routing can use baseline seating; FairPlay routing can call the backend router using predicted health.

E

Hourly room rollups next

Produce room_metrics_standard.json and room_metrics_fairplay.json-style outputs from playsim instead of static handcrafted fixtures.

07

Outputs and downstream use

Population mode exports a playsim-native hand corpus. The design favors rich evidence first; legacy projections can be added later if another consumer needs them.

meta

Run identity

Schema version, engine name, master seed, cap, data root, table count, total hands, and fixture note.

tables

Table summary

Per-table seed, horizon, hands played, seated count, and hands dealt by player.

hands

Rich hand docs

Seats, hole cards, board, actions with latency, results, pot size, and table id.

player_index

Player coverage

Hands dealt, hand target, and table ids per player. Useful for validating quotas and fixture coverage.

features

Scoring-ready rows

VPIP, PFR, aggression factor, timing regularity, soft-play delta, net BB, and archetype annotations.

Compression

Large but compressible

Use .gz output paths and --compact for CI or artifact handoff. The hand corpus is repetitive JSON and compresses well.

08

How to run each mode

Use Docker for the fixture-backed population command because it mounts the repo data root at /fixture. Use direct Python when you want arbitrary rosters.

Population fixture

All or selected fixture tables

cd playsim
docker compose run --rm playsim population \
  --data-root /fixture \
  --seed 42 \
  --cap 400 \
  --out /data/playsim_hand_histories.json.gz \
  --features /data/sim_player_features.json.gz \
  --compact

# Smaller subset:
docker compose run --rm playsim population \
  --data-root /fixture --seed 42 --cap 40 \
  --tables T-11 \
  --out /data/smoke_hands.json
Free-form session

Outside players.json

from playsim.runner import Player, run_session

roster = [
    Player(1, "new"),
    Player(2, "regular"),
    Player(3, "grinder"),
]

result = run_session(roster, 100, seed=42)
Decision summary

What this means operationally

Use population when the output should honor the Day-2 fixture population and table roster. Use run_session when you need controlled experiments, new synthetic rosters, calibration, or tests. Use routing today for a fixed-roster FairPlay-vs-Standard counterfactual; build Phase D before claiming room-scale live routing from the population simulator.

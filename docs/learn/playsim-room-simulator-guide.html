FairPlay Simulation Lab

How playsim's room simulator works — and what it found

A learning tool for the team: how synthetic poker players arrive, get seated, play, bleed, and leave a simulated room — and why, when we compared FairPlay health-routing against a plain "most-full" baseline, the baseline won. Each section is a place to stop and ask your own questions.

01

What this is

Playsim deals synthetic No-Limit Hold'em so that the signals our detector watches — VPIP, aggression, soft-play, robotic timing, chip bleed — emerge from play instead of being typed into a fixture. The room simulator is the newest layer: a closed loop that lets a routing policy decide where each arriving player sits, then measures the consequences.

The question

Does routing help?

FairPlay's pitch: steer vulnerable players to healthy tables so they survive longer and keep playing. We built this to test that — not assume it.

The trick

Decisions, not fixtures

Old comparison used two hand-written rosters (tested compositions). Now a policy makes the seating decision live, so the outcome is caused by the policy.

The honesty rule

Never grade your own homework

Routing uses predicted health from the scorer; the realized outcome is measured independently from chip flow. The two never touch. (§08)

02

The closed loop, end to end

One run = one room over a horizon (default 8 hours). The same seeded arrival stream feeds both A/B arms; only the seating policy differs. Walk the five stages, then dive into any of them below.

InputRoom state

12 tables seeded from table_roster.json (~68 seated at hour 0); ~54 unseated players become a seeded arrival stream.

DecidePolicy seats the seeker

Each arrival is placed by the active policy (most-full, random, FairPlay…). FairPlay asks the real backend router.

PlayHands are dealt

One hand per active table per step via PokerKit; chips move, the vulnerable cohort bleeds.

DecayPlayers leave

Losing shortens sessions (tilt model); tables that fall below 2 break and displace their players.

MeasureCausal trace out

Arrivals, routing decisions, seat events, sessions, hourly rollups, summary → room_sim_*.json.

North-star metric: retained paid seat-time for the vulnerable cohort (new, recreational, promo_hunter) — minutes accrued per hand dealt while seated at a live table. It is a sum, so it rewards keeping more vulnerable players seated and dealing.

04

Arrival → table

Demand is generated once and shared by both arms, so the only thing that differs is where each seeker is placed. playsim/playsim/arrivals.py + the policy seam in policies.py.

The stream

One intent per unseated player

The ~54 players not seated at hour 0 each get exactly one ArrivalIntent(player_id, archetype, arrive_at_min), drawn uniformly over the horizon from a seed. Flat schedule — no health-modulated arrival rate (that would make the two arms see different demand).

The decision

Policy picks a seat — or the seeker balks

The loop hands the seeker + the live table set to policy.choose(). If no acceptable open seat exists, the seeker balks (leaves without sitting, zero seat-time). A player arrives once; balking is terminal.

"Balk" = a seeker who arrives, finds no acceptable open seat, and leaves without playing. It's the queueing term, and it's the dominant lever in the findings — a balked vulnerable player contributes nothing to retained seat-time.
◆ Questions to investigate
  • Arrivals are uniform over the horizon. Real lobbies have rush hours. How would a time-varying (but still policy-independent) arrival curve change saturation and balking?
  • A player arrives exactly once. Should players re-queue later in the night? What would that do to the liquidity dynamics?
  • The room starts ~95% full. How sensitive is everything to initial occupancy? (See the findings — liquidity is the binding constraint.)
05

Playing a hand

PokerKit is the referee (legal actions, betting, showdown, chip accounting); we control the seeded deck so it's reproducible. The per-hand economics live in one shared helper — apply_hand_accounting in runner.py — used by both the single-table loop and the room loop, so they can't drift.

1 · Deal

Button rotates

One hand per active table per step; the dealer advances each hand. Agents act from an observation (hole cards, board, pot, position).

2 · Skill edge

A small honest nudge

Heuristic agents don't reliably out-win weaker players (variance dominates), so a small zero-sum skill_edge (default 1.6) transfers EV from weak to strong by skill gap. It's a labeled input; health stays derived.

3 · Account

Chips, busts, seat-time

Payoffs applied, stacks below 8bb rebuy (40bb start), seat-minutes accrue per dealt hand. This is the chip flow everything downstream is measured from.

Honest caveat baked in: the skill_edge transfer is a deliberate modeling choice — without it, cards + variance drown the predation signal. A trained solver brain (the solver_like/RLCard path) would make the edge emerge instead. Per-seed results are noisy; we always average over seeds.
◆ Questions to investigate
  • How much does skill_edge=1.6 drive the predation→decay signal? Sweep it and see where the routing story appears or vanishes.
  • 40bb starting stacks + rebuy-at-8bb means fast busts. Try deeper stacks — does survival stretch enough for health-routing to matter?
06

When players leave

This is the heart of the north-star metric. The vulnerable cohort logs off when its (loss-shortened) session budget is spent. Losing fast shortens the session — so an unhealthy table directly produces less retained seat-time. Shared helper cohort_should_leave in runner.py.

# a cohort player's session budget, in minutes
loss100 = (-net_bb) / (hands_played/100)   if hands_played >= 15 else 0   # warmup
tilt    = 1.0 - max(0, loss100)/130 * (0.5 + tilt_quit)                  # losing → shorter
tilt    = clamp(tilt, 0.12, 1.25)
budget  = base_session_min(archetype) * spread(0.85..1.15) * tilt        # spread = per-player jitter
#   → the player leaves once  seat_minutes >= budget
What shortens a session

Loss rate × tilt sensitivity

Faster bleed (high loss100) and a higher tilt_quit knob both shrink the budget. A new player (tilt-quit 0.80, base 20 min) at a predatory table evaporates; a grinder (tilt-quit 0.05, base 325) grinds on.

Why it's load-bearing

Decay is a consequence, not an input

loss100 comes from real played chips, so play-decay is a genuine downstream effect of who the player was seated with — exactly what routing is supposed to influence.

◆ Questions to investigate
  • The loss rate is only counted after a 15-hand warmup. Players who bust inside 15 hands never register a loss rate — does that mask early predation?
  • The tilt curve divides by 130 bb/100 and weights by 0.5 + tilt_quit. Are those constants doing too much? Try recalibrating against a target session-length distribution.
  • Only the cohort leaves voluntarily; the field stays put (to keep the experiment controlled). What breaks if predators/regulars also churn?
07

Table breaks & displacement

A table needs ≥2 players to deal. When departures drop a table below that, it breaks — and its remaining player is displaced. This turned out to be the mechanism behind the whole result (§11).

Break

Below quorum → table closes

Any table with exactly 1 seated player breaks that step; the lone player re-enters the seeker pool once and is re-routed by the active policy.

Re-seek guard

No phantom seats

A displaced player won't be seated at an empty table (that would just be a non-dealing lone seat); if no pair-forming seat exists, they balk. This keeps the eval trace clean.

Why this matters so much: every break displaces a player, and many displaced players can't re-seat (a "break-balk") and lose all further seat-time. Policies that keep tables full break fewer tables. That single fact decides the comparison.
◆ Questions to investigate
  • Real rooms actively balance tables (merge short games) to avoid breaks. Should the sim model an operator-side balancing step, separate from player routing?
  • Displaced players re-seek once then balk. Is that too harsh? How much seat-time is lost to break-balks specifically? (The findings instrument this.)
08

Predicted vs realized health (the guardrail)

Two different health measures, kept strictly apart — the spine of the whole design's credibility.

Predicted · drives routing

Composition-driven, top-down

The backend scorer (backend/scoring/health.py) scores a table from who is seated — predation pressure, fragility, cluster severity. Run with P_bleed=0 (no realized history), so it's purely compositional. FairPlay routes on this.

Realized · evaluation only

Chip-flow, bottom-up

After the fact, playsim measures what actually happened — how fast the cohort bled, busts, winner concentration. Never feeds a routing decision (a structural test enforces room.py can't import it).

Why split them? If the same formula both chose the seats and graded the outcome, "routing improved health" would be a tautology. Predicted decides; realized judges. The one cross-package seam that reaches the backend is playsim/playsim/router_adapter.py.
◆ Questions to investigate
  • The router rank is 0.30·Fit + 0.40·Health + 0.30·ΔHealth (frozen). Does the ΔHealth term ever push a fish toward a predator table because it "improves" that table most? (We probed this — it doesn't, given a clean choice — but worth re-checking under load.)
  • Predicted health holds P_bleed=0. Would giving the router some realized signal help, without creating circularity?
09

The policies

All five live behind one interface (policy.choose(seeker, live_tables)) in policies.py; the room loop is policy-agnostic.

PolicyRuleBackend?Status
most-full (Standard)Seat the fullest table with an open seat (liquidity-seeking — what real rooms do)nobaseline
randomUniformly random open table (neutral "no intelligence" control)nobaseline
fairplay-routeBest non-gated open table by the real frozen router's rankyesheadline
fairplay-protectRoute, but defer a vulnerable seeker when only sub-threshold seats are openyesexperimental
fairplay-balancedSpread fish across healthy-enough tables (fewest fish first)yesbackfired
Gated tables (a seated high-band cluster) are removed from a vulnerable seeker's candidate set entirely — the integrity hard-gate. The player could still pick it manually; the policy just won't recommend it.
10

How we compare (the A/B design)

A clean counterfactual: identical starting room, identical seeded arrival intents, identical economics — only the seating policy changes. Placements and departures then diverge because of the policy, which is the whole point.

Shared

Demand & seeds

Same arrival stream, same per-table seeds. Both arms see identical "who wants a seat, when."

Derived

Placements & departures

Where each seeker lands, who bleeds, who leaves, which tables break — all flow from the policy's choices.

Honest

Averaged over seeds

Single-seed runs are noisy; the headline is a mean over a pinned seed set. The directional sign is reported, not asserted.

Everything is deterministic: same (seed, horizon, policy) reproduces byte-for-byte. Each table owns one persistent RNG for its whole life, so replay survives even a mid-run table break.
11

Findings: Standard wins (and why)

Four policies, matched 4-hour and 8-hour horizons, all 12 tables, 3 seeds averaged. Headline = vulnerable-cohort paid seat-hours; we also instrument table breaks and break-balks.

4-hour horizon (240 min)

What happened

Standard wins or ties; FairPlay fades with time

most-full ≈ random ≈ fairplay at 4h; by 8h fairplay is ~9% behind. More time made health-routing worse, refuting the "give survival time to express" hypothesis. Random ≈ Standard, so Standard isn't an unfairly strong baseline.

Why — verified

Table liveness, not a bad decision

A router probe shows the decision is sound (it picks the healthy table, rank 57 vs 35). The loss is churn: health-routing scatters players → more tables drop below quorum → more breaks (38 vs 28) → more break-balks (34 vs 24) → seat-time lost. most-full concentrates and keeps tables alive.

The fix that backfired: fairplay-balanced deliberately spread fish to de-concentrate — and was the worst arm, with by far the most breaks (≈48). Spreading thin fragmented the room below quorum. Strong confirmation that at this scale, liquidity (concentration) beats distribution.
Read this as: a plausible product intervention measurably backfired for a non-obvious systems reason — which is exactly what an honest simulator is for. It is not a verdict that risk-aware routing is bad in a real room. Caveats: synthetic agents, 40bb stacks, near-full room, a throughput-shaped metric; we did not tune toward a positive result.
12

Open questions for the team

The most useful next experiments — pick one and the harness in §13 will run it.

Metric

Is throughput the right yardstick?

Cohort seat-time is a sum, so it rewards liquidity. Try a per-player survival or "bad sessions prevented" metric — FairPlay's value is harm-avoidance, not seat count.

Scale

Does a less-full room flip it?

At ~95% occupancy, liquidity is the binding constraint. With more open seats, the health benefit might finally show.

Policy

Liveness-aware FairPlay

Add a quorum/liquidity term to the rank so health-routing stops stranding tables — unlike the naive spread that backfired.

Operator step

Model table-balancing separately

Real rooms balance tables independently of player routing. Should the sim too?

Agents

Real brains

Swap heuristic agents for a trained solver_like/RLCard brain so the skill edge emerges instead of being injected.

Leave model

Recalibrate tilt

The 15-hand warmup and tilt constants are uncalibrated. Fit them to a target session-length distribution and re-run.

13

Run it yourself

Everything is seeded and reproducible. From the playsim/ directory:

make install                      # one-time: Python 3.12 venv + PokerKit + editable install

# The N-way comparison behind the findings (random / most-full / fairplay / fairplay-balanced)
.venv/bin/python analysis/room_policy_comparison.py --horizon 240 --seeds 42,7,99
.venv/bin/python analysis/room_policy_comparison.py --horizon 480 --seeds 42,7,99

# The shipped Standard-vs-FairPlay A/B → writes room_sim_*.json + room_metrics_*.json
.venv/bin/python -m playsim.cli room-sim --seeds 42,7,99 --horizon 480 --out-dir out

# --debug-trace adds the full ranked candidate list to every routing decision (verbose)
.venv/bin/python -m playsim.cli room-sim --seeds 42 --horizon 480 --out-dir out --debug-trace

make test                         # full suite
Tip for fast local pokes: add --tables T-22,T-8 and a short --horizon 60 to iterate in seconds before committing to a full run.
Reading "why this table?": every backend-routed entry in routing_decisions carries a compact rationale (rank, health, delta_health, seating_risk, badge, integrity_gated) by default. --debug-trace adds the full ranked candidate list; per-hand internals live in the separate playsim_hand_histories_* artifact, never inlined here.

1.Plain-English Explanation

What playsim is. A deterministic, seeded poker simulator. Archetype-driven agents play real No-Limit Hold'em hands on the PokerKit engine, and the behavioral + integrity signals (VPIP, PFR, aggression factor, timing, soft-play, net_bb) emerge from play rather than being hand-set (playsim/README.md:1-28). Critically, playsim is not the detector — risk scoring lives in backend/scoring/ (README:10-14), and no LLM is in the loop.

What it runs today. Eight CLI subcommands are wired (playsim/playsim/cli.py:225-307): run (integrity-loop hand generation), routing (fixed-roster Standard-vs-FairPlay backtest), population (fixture-population replay), room-sim (the new closed-loop room A/B), plus replay (determinism check), calibrate, health, and tables.

What the closed-loop room simulator adds. The earlier routing mode compared two static, human-written rosters — which the findings doc itself calls "nearly tautological." The new RoomSim (playsim/playsim/room.py:498-518) steps a global clock and makes the seating policy earn the seat-time: it seats due arrivals, deals a hand per quorum-met table, checks tilt departures, and handles table breaks each step. It is "closed-loop" because state feeds the next decision — departures reopen seats, arrivals are seated against the live table composition, and tables that fall below two players break and displace their occupants. Both A/B arms share the same seeded arrival stream; only the seating policy differs (test_arrivals.py:26-30).

What the room-routing findings say. The headline: the Standard "most-full open table" policy wins or ties; FairPlay-route (health-aware) falls roughly 9% behind by hour 8; Random tracks Standard; and FairPlay-balanced is the worst. Doc figures (findings.md:99-104): at 8h, random 9.94 / most-full 10.06 / fairplay 9.16 / balanced 8.62 cohort seat-hours.

Illustrative, not validated
The behavioral model runs on hand-authored parameters, so every retention number the room simulator produces today is illustrative (docs/learn/playsim-calibration-data.md:12-19). The metric magnitudes are mechanically produced by real code but not pinned by any committed snapshot. The decision direction is testable; the numbers are not yet claimable.
Implemented / verified
The simulator engine, all five policies, the backend routing seam, the closed-loop room loop, the predicted-vs-realized health separation, and the fit-aware behavior model (default-off) are all built and tested for well-formedness and determinism. The machine is buildable-and-built; it is the calibration that is missing.

(A) Is the thesis testable with this simulator?

Yes — structurally. The thesis ("health-aware routing protects table health / recreational players better than naive liquidity-seeking") is testable here precisely because the simulator separates the variable that chooses seats from the variable that grades them. Predicted health (composition-only, sessions=None, P_bleed zeroed at backend/scoring/health.py:206-207) drives routing; realized chip-flow health (playsim/playsim/health.py:24-44) grades outcomes and never feeds routing — enforced structurally by a no-import test (test_room.py:133-139) and a P_bleed==0 assertion (test_room_adapter.py:50-54). That independence is what makes Standard-vs-FairPlay a real test rather than a tautology. But the thesis is only quantitatively testable once the behavioral parameters are calibrated to data; today it is testable in direction, not magnitude. The fit-aware behavior work (Phase 2, done but default-off) is the piece that even lets the thesis's central variable (Fit) buy retention — without it the old leave model responded only to chip loss, so "the simulator cannot reward the exact variable the thesis is built on" (brainstorm 2026-06-24:28-32).

3.Strengths

  • Emergent signals, not hand-set outputs. Behavioral and integrity stats fall out of actual PokerKit hands (README:1-28) — the simulator is harder to accuse of begging the question than a stat-generator would be.
  • The anti-circularity guard is structural, not aspirational. Predicted health routes; realized health grades; the two cannot touch — and a test introspects room.py's source to forbid the import (test_room.py:133-139), with P_bleed==0 asserted (test_room_adapter.py:50-54). This is the single most valuable design property here.
  • Fair A/B construction. Both arms consume the identical seeded arrival stream with no health-modulation (test_arrivals.py:26-30,63-67). The policy is the only moving part.
  • One backend seam, single source of scoring truth. Only router_adapter.py imports scoring.*; playsim never re-implements routing. Backend stays the one place scoring is defined.
  • Honest baseline + honest labeling. "Most-full open table" is a realistic operator null, and the findings doc volunteers its own weaknesses (throughput-shaped metric, no manufactured FairPlay win — quoted in §5).
  • Phased, parametric behavior model that genuinely degrades to the old one. FitAwareBehaviorPolicy is real, not stubbed: at w_pressure=w_fit=0 with decline off it reduces exactly to the default (behavior.py:159-161) — a clean, testable extension point.
  • Determinism discipline. Room, canonical, and comparison runs reproduce exactly on re-run; single tables are byte-identical including boards/payoffs (test_determinism.py:7-13).
  • Provenance-before-feature for deferred work. RLCard is correctly scaffolded-and-deferred (baselines.py:86 raises NotImplementedError; agent.py:55-56 carries agent_model/version metadata now).

4.Concerns / Risks (ranked, with cheapest test)

This ordering folds in (D) ranking of simulation-design risks: the highest-ranked items are the ones that most threaten any quantitative conclusion (authored mechanism → headline; metric shape; seed stability), followed by seam-fragility and artifact-integrity risks.

5.Findings Review

What the findings prove. That in this synthetic model, at this scale, under these authored parameters, the most-full policy accumulates the most cohort paid seat-time, FairPlay-route trails, and FairPlay-balanced is worst. The producing logic is real and inspectable: room_policy_comparison.py:38-93 provably computes the reported metrics over seeds 42,7,99 with equity 6 across all 12 tables. The mechanism is also present in code, not just prose: tables breaking at the <2 quorum (room.py:420) and re-seek-once-then-balk (room.py:428-436) mean most-full keeps tables above quorum while health-scattering creates more near-quorum tables → more breaks → more balks. "Most-full keeps tables alive, health-routing scatters them into breaks" is a genuine code property.

What they do NOT prove. (1) The exact magnitudes — no committed snapshot or test pins 9.16 vs 10.06; the label is illustrative-but-mechanically-produced. (2) That the gap exceeds seed variance (Risk 3). (3) Anything about real poker rooms. (4) The router-probe numbers (predator-nest health 31.7 vs balanced 90.0; rank 57.0 vs 35.2, findings.md:121-123) have no producing script or test — as stated numbers they are UNSUPPORTED; only the decision direction is testable (test_room_adapter.py:71-95).

Are the conclusions honestly labeled? (quoting the labels)

"this is a result about THIS synthetic model at this scale, not a verdict on real poker rooms" — findings.md:28-29
"The metric is throughput-shaped. Cohort paid seat-time is a sum, so it structurally rewards liquidity. A per-player-survival or harm-avoided metric might rank the policies differently" — findings.md:164-166
"We deliberately did NOT tune the model to manufacture a FairPlay win" — findings.md:170-171
"Every retention number the room simulator produces today is illustrative, not validated, because the player behavioral model runs on hand-authored parameters" — calibration-data.md:12-19
"This is not a blocker to BUILDING (the model and harness are done); it is the blocker to CLAIMING" — calibration-data.md:100-102

The labeling is unusually honest — the doc indicts its own metric and refuses to claim the numbers. This is a credibility asset, not a weakness.

(C) Stress-test — what could flip or weaken the findings

  • Flip the metric → likely re-rank. The headline is a throughput sum; the doc itself says a per-player-survival metric "might rank the policies differently." Because most-full maximizes the chosen metric by construction, switching to harm-avoided is the single most likely thing to overturn the headline. This is the highest-leverage stress test (Risk 2).
  • Turn on fit-aware behavior → gap should narrow. FairPlay is currently judged with its Fit benefit disabled (Risk 4). With FitAwareBehaviorPolicy on, well-fit seats earn retention — the mechanism FairPlay routes for. If the gap narrows or flips, the headline was an artifact of the disabled lever; if it doesn't, that's stronger evidence against health-routing helping (and the sweep self-warns that an edge appearing only at high fit-weight is itself an artifact).
  • Perturb the authored constants → possible flip. Quorum=2, re-seek-once, the tilt curve — these three constants drive the break/balk mechanism that produces the gap. A sensitivity sweep is the cheapest way to learn whether the ranking is robust or parameter-bound (Risk 1).
  • Seed variance → partly resolved by reproduction. Reproduced (3 seeds 42/7/99): at 8h the gap is seed-stable — Standard > FairPlay on all three seeds and the means reproduce the doc exactly. But at 4h it flips per-seed (FairPlay wins seed 42, loses seed 7), so the early-horizon gap is within noise. Still no ranking assertion committed to the new loop, and a ≥20-seed mean±std would pin the 8h result rigorously (Risk 3).
  • The anti-circularity conclusion itself rests on unreproduced sweep numbers. The guard is structurally real, but "anti-circularity passed" depends on sweep outputs that aren't pinned by a committed snapshot. Good discipline; not yet evidence.
Net read on the findings (updated post-reproduction)
Honestly framed and mechanically genuine. Reproduction did not shrink the numbers — the doc's 4h and 8h means reproduce to the decimal, and the 8h Standard>FairPlay gap holds on all three seeds — so the result is real and faithfully reported, not a single-seed artifact (4h, however, is noisy and flips per-seed). What remains open is interpretive, not numeric: the gap is measured under a throughput-shaped metric that most-full maximizes by construction, with FairPlay's fit lever switched off, in an uncalibrated model. So treat "Standard beats FairPlay" as "under a throughput metric with the fit lever off, in this uncalibrated model" — a true and reproducible statement about this configuration, not a verdict on routing.

6.Questions Teammates Are Likely To Ask

Why not just use LLM agents?
Because determinism is a product rule and an LLM player is just another ungrounded prior — "intelligence ≠ fidelity" (brainstorm 2026-06-24:65-69). A seeded archetype agent is reproducible and inspectable; an LLM would make the A/B non-deterministic and unfalsifiable without buying any real-world fidelity.
Why call the backend at all — why not score in playsim?
To keep a single source of scoring truth. router_adapter.py is the only module importing scoring.*; the real route() (backend/scoring/router.py:95-148) is exactly what production would use. Re-implementing scoring in the sim would let the two drift and would make the test grade a different scorer than the one shipping.
Predicted vs realized health — what's the difference and why two?
Predicted (backend, composition-only, sessions=None, P_bleed=0 at health.py:206-207) chooses seats. Realized (chip-flow, playsim/playsim/health.py:24-44) grades outcomes and never feeds routing (enforced by no-import test test_room.py:133-139). Two of them is the whole point: if the same signal chose and graded, the A/B would be a tautology.
Why is the v1 room_metrics_* schema not enough?
It's a flat 8-row presentation projection — great for charts, useless as a causal trace. It drops every routing decision, arrival intent, seat/break event, session, and the eval summary; collapses predicted-vs-realized; and fabricates fields (projected_eod_*, hour_note, reward_fee_ratio=0.0). The FairPlay value prop ("routed P-104 to T-8 not T-22 because rank/health/delta/gated…") lives only in canonical room_sim_* (room_export.py:84-120).
Can we simulate players who aren't in players.json?
Yes — that's what run_session (runner.py:176) is: it takes any Player list. Population mode is the fixture-bound replay (hour-0 roster only, no routing); room-sim and population are both built on the free-form engine, so synthetic non-fixture players are supported at the engine level.
What makes it deterministic?
Seeded RNG threaded through the engine, identical seeded arrival streams across both A/B arms (test_arrivals.py:26-30), and value-level reproduction tests (room/canonical/comparison reproduce on re-run; single tables are byte-identical including boards/payoffs, test_determinism.py:7-13). Caveat: reproduction is verified in-process, not yet disk-byte-level across a fresh process (Risk 9).
Why does Standard beat FairPlay so far?
A real code mechanism: tables break below 2 players (room.py:420) and displaced players re-seek once then balk (room.py:428-436). Most-full keeps tables above quorum; health-routing scatters players across tables → more near-quorum tables → more breaks → more balks → less seat-time. Plus the metric is a throughput sum that most-full maximizes by construction.
Does that mean FairPlay is wrong?
No. It means (1) the metric is throughput-shaped, not protection-shaped (the doc says so, findings.md:164-166); (2) FairPlay's main lever — Fit-driven retention — is currently switched off (behavior.py:150-204 default-off); and (3) nothing is calibrated. Under a harm-avoided metric with fit-aware behavior on, the ranking may well diverge. "FairPlay loses" is not yet a claim the simulator supports.
What data would make this validated rather than illustrative?
Real behavioral ground truth to calibrate w_fit, w_pressure, and the tilt curve (Phase 3 — the documented gate). The harness exists (fit_sensitivity_sweep.py) but the actual fit-to-data calibration is unimplemented. Until then: "the model and harness are done; it is the blocker to CLAIMING" (calibration-data.md:100-102).

7.Verdict

Is the approach sound? Yes. The architecture is the right one for this thesis: emergent signals, a single backend scoring seam, a fair seeded A/B, and — most importantly — a structurally enforced separation between the health that routes and the health that grades. That separation is what elevates this above a self-fulfilling demo. The honesty discipline in the findings doc (indicting its own metric, refusing to claim numbers, deliberately not manufacturing a FairPlay win) is exactly what you want from a research artifact.

What to build next (in priority order):

  1. Re-score existing canonical runs under a per-player-survival / harm-avoided metric (Risk 2) — the single highest-leverage, cheapest test; the fields already exist in summary.
  2. Run the A/B with fit-aware behavior on at documented weights (Risk 4) — exercise the lever FairPlay actually routes for.
  3. Add a many-seed stability assertion to the new room loop (Risk 3) — reproduction confirms the 8h gap holds on 3 seeds and reproduces the doc exactly; the 4h gap flips per-seed. A committed ≥20-seed mean±std assertion would lock this in.
  4. Harden the backend seam: keyword args, import (not copy) VULNERABLE_ARCHETYPES, one contract test (Risk 5).
  5. Regenerate the checked-in room_metrics_* from the simulator and stop shipping hand-authored fixtures as computed (Risk 6).
  6. Serialize the behavior block into run_config — done in PR #47 (room_export.py:125-126); what remains is a disk-byte / reproduce-from-artifact round-trip test and capturing the behavior seed as a scalar (Risks 8, 9).

What should NOT be claimed yet:

  • Any numeric retention/seat-time magnitude (9.16 vs 10.06, the ~9% gap) — illustrative until calibrated.
  • "Standard beats FairPlay" as a general statement — it is "under a throughput metric, fit lever off, uncalibrated, this synthetic model, this scale."
  • The router-probe numbers (31.7 / 90.0 / 57.0 / 35.2) — no producing script; UNSUPPORTED as stated.
  • "FairPlay is wrong / health-routing doesn't help" — the test that would fairly judge it (fit-aware on, harm-avoided metric) hasn't been run.
  • "Byte-identical replay" at disk level — only value-level/in-process reproduction is tested today.

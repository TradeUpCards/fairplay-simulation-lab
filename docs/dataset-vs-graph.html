Raw property graph vs. the rolled-up findings view

Two artifacts, two layers of one system. The kgonto ontology (this repo) describes the raw property graph — atomic nodes and the derived edges detection runs on. The fairplay-simulation-lab/data fixtures are a rolled-up findings view — a deliberate, post-detection summary shaped for the demo and the PRD's worked numbers. This doc reconciles the two and defines the contract between them.

Correction to the earlier version of this page. The first pass compared the sim-lab fixtures against the raw ontology and flagged aggregations as "missing edges / wrong shape." That was an unfair comparison: the fixtures are a summarized view of case findings, presented that way on purpose for the demo — not raw graph data. A findings view is supposed to collapse thousands of pairwise edges into a rate and pre-resolve cases to escalate/monitor/benign. The right question isn't "what's missing from the dataset" — it's "what does each layer own, and does the rollup preserve what the demo actually needs?"

Layer 1 — Raw property graph this repo

system of record · detection substrate

Granularity: one node per real entity (each Account, Device, Seating event); one edge per observed relationship; derived pairwise edges (CO_SEATED_WITH, SOFT_PLAYS_AGAINST) materialized by traversal.

Shape: nodes + typed edges, built to be traversed — WCC for components, convergence counting per component, Evidence nodes per reason code.

Audience: the detection engine. Not meant to be read by a human.

Layer 2 — Rolled-up findings view sim-lab/data

demo fixture · post-detection summary

Granularity: one object per finding — a cluster, a household, a device group — each holding a member_ids[] array and pre-computed aggregates (co_seating.rate, soft_play_delta).

Shape: compact JSON built to be rendered — cases already resolved to escalate/monitor/benign, signals pre-summarized, ground-truth labels attached.

Audience: the demo, the PRD numbers, the eval panel.

So the aggregations are correct. device_groups holding a player_ids[] array, co_seating as a single rate, soft_play_delta as one number per player, cases pre-labeled — these are exactly what a rolled-up view should look like. They are not deficiencies in the data; they are the rollup doing its job.

05The 3 intended compressions

The earlier page called these "structural mismatches." They're actually the projection rules — deliberate compressions that turn the raw graph into a demo-readable view. Each is correct; the note flags where it goes a step too far.

① Edge fan-out → group object

USES_DEVICE → device_groups · MEMBER_OF → member_ids[]

N pairwise edges collapse into one container holding the accounts. Correct & lossless — the group can be re-exploded into edges any time. The demo never needs the fan-out.

② Pairwise edges → aggregate stat

CO_SEATED_WITH → co_seating.rate · soft-play → delta

Per-pair edges roll up to one rate/score per finding. Lossless for co-seating (the demo shows the rate). Lossy for soft-play/outsider — collapsing to a per-player scalar / cluster boolean drops who did it to whom, which the convergence story leans on.

③ Cases pre-resolved → status + prose

convergence → escalate/monitor/benign · evidence → note

The raw graph's WCC + convergence count is already computed in the view; the result is a label plus an explanatory note. Correct — except the explanation (esp. counter-evidence) is free text, not a structured field the UI/GraphRAG can key off.

The pattern

All three are the same move: the raw graph holds the atomic relationships and runs detection; the findings view ships the conclusions. Two of the three are perfectly faithful. The compression only bites where the demo needs a detail the summary threw away — which is exactly the short list in §06.

07Who owns what

The division of labor that keeps the two repos consistent.

This repo (kgonto) — Layer 1

  • Owns the raw-graph ontology: atomic nodes, raw edges, derived pairwise signal edges, Evidence nodes.
  • Owns the detection logic: WCC / Louvain / convergence gate that produces findings.
  • Is the system of record — the source the findings view is projected from.
  • In production, generates the rollup continuously; for the capstone it's the schema the demo fixture conforms to.

sim-lab/data — Layer 2

  • Owns the findings view: pre-resolved cases, aggregated signals, ground-truth labels.
  • Is a hand-authored static fixture for the demo (Day-2) — legitimately so; it stands in for the rollup until detection runs live.
  • Should carry the 3 demo-critical fields from §06 so the UI/eval don't parse prose.
  • Plus the pure-tabular room metrics — eval/ROI, never graph data.
The contract between them. Define the projection raw-graph → findings-view once (which raw structures collapse into which fixture fields). Then the static demo fixture can later be swapped for a generated rollup with no change to the consumers. The build plan's "static fixtures first, computed second" is exactly this swap.

08Raw-graph schema (Layer 1) & its projection

The raw graph this repo owns — and, in comments, how each piece rolls up into the findings view. This is the collusion-core slice; the full production schema is the four-layer taxonomy in docs/10-production-graph-schema.md (adds Customer / IP / Location / Payment nodes, ENTITY_MATCHES / CHIP_FLOW_TO / IMPACTS_TABLE edges, and CONTRADICTED_BY counter-evidence).

// ---- LAYER 1 RAW + LAYER 2 DERIVED (this repo · collusion-core slice of doc 10) ----
Customer {customer_id, kyc_level, status}                 ⟶ not in the rollup (identity spine)
Account  {account_id, archetype, risk_score, vpip, pfr, ...}
Device · IP · Location · PaymentInstrument          ⟶ Device ⟶ device_groups; IP/Loc/Pay new at prod
Seating  {seating_id, seated_at, left_at, paid_seat_time}
Cluster  {case_id, status, integrity_score}
Evidence {evidence_id, evidence_type, score, confidence, is_counter_evidence}

// raw (observed facts)
(Account)-[:BELONGS_TO_CUSTOMER]->(Customer)     ⟶ implied by cluster member_ids[]
(Account)-[:USES_DEVICE]->(Device)            ⟶ device_groups.player_ids[]
(Account)-[:SEATED_AT]->(Seating)-[:AT_TABLE]->(Table)  ⟶ sessions[] rows

// derived signals — carry window_start/end, score, confidence, support_count, evidence_ids
(Account)-[:SHARES_DEVICE]-(Account)            ⟶ edge_type:"device_link"
(Account)-[:ENTITY_MATCHES {match_score}]-(Account) ⟶ not in rollup (ER verdict)
(Account)-[:CO_SEATED_WITH {rate}]-(Account)      ⟶ co_seating.rate (aggregate, faithful)
(Account)-[:TIMING_CORRELATED]-(Account)         ⟶ kept as edges in view
(Account)-[:SOFT_PLAYS_AGAINST]->(Account)       ⟶ soft_play_delta scalar (lossy)
(Account)-[:TARGETS_OUTSIDER]->(Account)         ⟶ outsider_pressure_signal bool (lossy)
(Account)-[:CHIP_FLOW_TO {net_amount}]->(Account)   ⟶ not in rollup (AML / dumping)
(Cluster)-[:IMPACTS_TABLE {health_delta}]->(Table)  ⟶ high_casual_exit_impact bool

// case (analyst workflow)
(Account)-[:MEMBER_OF]->(Cluster)
(Cluster)-[:SUPPORTED_BY]->(Evidence)         ⟶ member_ids[] + note
(Cluster)-[:CONTRADICTED_BY]->(Evidence)      ✓ FIXES §06 — counter-evidence now structural
(Cluster)-[:CONCERNS_TABLE]->(Table)         ⟶ case_ref / seeded_table_label

// ---- LAYER 2 FINDINGS VIEW (sim-lab/data) = projection of the above ----
// faithful: device_groups, co_seating, sessions, member arrays, cases→status
// add for the demo (§06): evidence[] incl. CONTRADICTED_BY, convergence breakdown, archetype, thresholds
Update. The §06 "counter-evidence is prose" gap is now resolved at the schema level: doc 10 makes CONTRADICTED_BY a first-class case edge and adds three *_counter_evidence reason-code types. The rollup just needs to carry them as an evidence[] field — no longer an open modeling question.

09Next steps

  1. Document the projection contract ½ day
    One table: each raw-graph structure → its findings-view field (the §08 comments, formalized). This is the seam that lets the static fixture become a generated rollup later.
  2. Add the 3 demo-critical fields to the findings view sim team · ½ day
    Structured evidence[] (incl. counter-evidence), per-signal convergence breakdown, explicit archetype + thresholds.json. Small, view-only — no ontology change.
  3. Build the raw graph in this repo 1–2 days
    Stand up Layer 1 from the simulator's raw events (not from the rollup): atomic nodes, raw + derived pairwise edges, Evidence nodes. This is the producer the view is projected from.
  4. Run detection on Layer 1, reproduce the findings 1 day
    WCC + convergence over the raw graph should re-derive CASE-C escalate / CASE-E monitor / CASE-D benign — i.e. regenerate the rollup. Asserting that match validates both layers.
  5. Wire validation to seeded_case_labels ½ day
    Every eval_checks entry becomes a passing assertion against the graph-derived findings — green means Layer 1 reproduces the ground-truth view.
  6. GraphRAG over the raw subgraph after build
    Anchor on Cluster, pull the 2-hop raw subgraph incl. structured counter-evidence → grounded pit-boss summary. Now possible because Evidence is structured at Layer 1.
One-line verdict. The sim-lab fixtures aren't a flawed dataset — they're a faithful rolled-up findings view of the raw graph this repo owns. The two layers are producer and projection. The only real work is (a) defining the projection contract, (b) adding three demo-critical fields the rollup currently buries in prose, and (c) standing up the raw graph here so the view can be generated from it rather than hand-authored.
